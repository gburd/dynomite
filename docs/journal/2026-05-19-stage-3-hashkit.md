# Stage 3 - Hashkit and tokens

Branch: `stage/3-hashkit`
Status: READY_FOR_REVIEW

## Summary

Stage 3 ports `_/dynomite/src/hashkit/` plus the contrib MurmurHash3
implementation. New artifacts:

* `crates/dynomite/src/hashkit/` - one module per hashing algorithm
  (`one_at_a_time`, `md5`, `crc16`, `crc32`, `fnv`, `hsieh`, `murmur`,
  `jenkins`, `murmur3`), plus `token`, `ketama`, `modula`, `random`, and
  the `HashType`/`hash` dispatch surface in `mod.rs`.
* `crates/dyn-hash-tool/` - new binary crate exposing the algorithms
  on the command line. The output format is one line per key,
  `<algorithm>:<key>:<token-hex>`, documented in the crate rustdoc.
* `crates/dynomite/tests/fixtures/hashkit_vectors.json` - 858 frozen
  conformance rows. Generated by
  `cargo run -p dynomite --example generate_hashkit_vectors`.
* `crates/dynomite/tests/stage_03_hashkit.rs` - reads the fixture and
  asserts every row reproduces. Includes RFC 1321 MD5 anchors and
  canonical FNV-1a-32 anchors that do not depend on the JSON corpus.
* `crates/dynomite/tests/stage_03_properties.rs` - proptest harness
  for `set_int_dyn_token`/`get_int` round-trip, `cmp` total-order
  invariants, deterministic dispatch, and parser totality.
* `crates/dynomite/tests/stage_03_distributions.rs` - end-to-end ring
  routing determinism, ketama topology-change locality, modula modulus
  agreement.
* `crates/dyn-hash-tool/tests/smoke.rs` - assert_cmd-driven CLI smoke.

## Parity diff

13 algorithm functions, 8 token functions, 3 distribution dispatchers,
and 1 RNG were added under `dynomite::hashkit::*`. Three deviations
recorded:

1. `hash_murmur3` is a no-op in C; Rust implements the real 128-bit
   algorithm against the contrib reference.
2. `hash_fnv1a_64` is preserved bug-for-bug (32-bit accumulator with
   32-bit prime) so token rings keep their existing routing.
3. `dyn-hash-tool.c` does not exist in the C tree; Rust defines its
   own `<algorithm>:<key>:<token-hex>` output.

## Open questions

None. The conformance corpus is anchored against:

* Canonical RFC 1321 MD5 vectors (empty, `a`, `abc`, `message digest`,
  the alphabet, the ASCII alphanumeric corpus).
* Canonical FNV-1a-32 vectors.
* The standard CRC-32 vector for `123456789` (`0xCBF43926`).
* MurmurHash3 x86_128 of the empty string with seed 0 = `[0,0,0,0]`
  (a property of the algorithm independent of seed choice).

The remaining hash-specific values come from walking the C source by
hand and locking the result into the JSON corpus. A reviewer who wants
to double-check can re-run the generator and diff.
