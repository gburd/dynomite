## What landed

The remaining three Section-6 follow-ups from
`docs/design/aae-comparison.md`:

* **R3** -- per-token sweep cadence so peers that hold N tokens
  exchange independently. Backward-compat: opt-in via
  `ConfAae::per_token_exchange`.
* **R4** -- operator-facing AAE counters and gauges in a new
  `aae/metrics.rs` module. The accumulator's shape mirrors
  `dynomite::stats::FailureMetrics`; cross-crate plumbing
  was deferred (the engine handle lives on the do-not-touch
  list this branch operates under) and the embedding wires
  the AAE handle directly via `Scheduler::install_metrics`
  and `RepairScheduler::with_metrics`.
* **R5** -- `dyn-admin aae-status` CLI subcommand backed by
  new PBC ops 220 / 221. Server-side the op is routed
  through a pluggable `AaeStatusProvider` trait; default
  (`NoopAaeStatusProvider`) returns an empty snapshot so
  unconfigured deployments do not error.

The shipped wire format and the existing exchange protocol
phases are untouched. None of R3 / R4 / R5 add a new
top-level dependency; the only Cargo manifest change is
adding `parking_lot` to `dyniak` (already in the workspace).

## R3: per-token exchange

### Config

```rust
// crates/dyniak/src/aae/config.rs
pub struct ConfAae {
    // ...existing fields...
    /// When `true`, the scheduler emits one sweep tick per
    /// `(peer, token, time_bucket)` triple instead of one
    /// per `(peer, time_bucket)` pair.
    ///
    /// Default `false` (per-peer cadence) so existing
    /// deployments are unaffected.
    pub per_token_exchange: bool,
}
```

### Scheduler additions

* `SweepTick` gains an `Option<String> token_hex` field. The
  hex form is the same string `DynToken::to_hex` produces;
  it stays cheap to clone and trivially serialisable.
* `SweepPlan::new_per_token(peers_with_tokens: &[(u32,
  Vec<DynToken>)], n_time_buckets: u32, cfg: &ConfAae)`
  emits one tick per `(peer, token, time_bucket)` triple.
  The plan walks peers round-robin within a token slot,
  then advances to the next slot, then advances the time
  bucket; this guarantees every peer sees a slice of
  progress before any peer's next-time-bucket sweep starts.
* `Scheduler::exchange_per_token(local: &Tree, remote: &V,
  tokens: &[DynToken])` runs an `Exchange` per token and
  pairs each token's hex with its divergence list. The
  helper exists so embeddings can drive per-token
  cadence without reimplementing the loop.

The shape of `Tree` itself is unchanged. Per-token granularity
is about *cadence and reporting*, not tree partitioning;
Riak's vnode-style sharding remains a future project (the
brief flagged this as an explicit non-goal for the slice).

### Tests

Five new scheduler tests:
* `per_token_plan_emits_one_tick_per_triple`
* `per_token_plan_handles_no_peers`
* `per_token_plan_handles_peers_with_no_tokens`
* `per_token_plan_token_hex_roundtrips`
* `exchange_per_token_runs_one_exchange_per_token`
* `exchange_per_token_surfaces_divergence`

(`LocalPeerView` gained `derive(Clone, Copy)` so it can be
re-used across token loops without rebuilding.)

## R4: AAE metrics

### Module: `crates/dyniak/src/aae/metrics.rs`

* `AaeMetrics` -- single-mutex accumulator (parking_lot,
  matching `FailureMetrics`).
* `AaeMetricsSnapshot` -- immutable view; `is_empty()`
  predicate so the embedding can skip rendering.
* Counters and gauges (the names match the `aae_*` family
  Riak operators expect and what
  `docs/book/src/operations/metrics.md` now documents):
  * `aae_exchange_attempts_total{peer_idx, dc, rack}`
  * `aae_exchange_success_total{peer_idx, dc, rack}`
  * `aae_exchange_divergent_keys_total{peer_idx, dc, rack}`
  * `aae_repair_dispatched_total{peer_idx, dc, rack}`
  * `aae_tree_segments_dirty_gauge{peer_idx}`
  * `aae_full_sweep_last_completed_seconds_gauge{peer_idx}`
  * `aae_snapshot_save_total`
  * `aae_snapshot_load_total`
  * `aae_snapshot_corruption_total`

### Wire-up points

* `Scheduler::install_metrics(handle)` plus
  `Scheduler::observe_exchange_attempt`,
  `observe_exchange_success`, `observe_divergent_keys`.
  The driver (in the embedding) calls these from the per-tick
  hot path; absent a metrics handle the calls are no-ops.
* `RepairScheduler::with_metrics(handle, dc, rack)`:
  every successful `Outcome::Repaired` `resolve_all`
  emits adds to `aae_repair_dispatched_total`.
* `metrics::save_snapshot_with_metrics(tree, path, m)` /
  `load_snapshot_with_metrics(path, m)`: thin wrappers
  around the existing `Tree` snapshot calls. The save
  helper bumps `aae_snapshot_save_total`; the load helper
  bumps `aae_snapshot_load_total` on success and
  `aae_snapshot_corruption_total` on the `Corrupted`,
  `VersionSkew`, or `BadShape` variants of `PersistError`
  before re-raising. `Io` (the common `NotFound` "no prior
  snapshot, rebuild" path) does not count as corruption.

### Why a new module instead of `dynomite::stats::FailureMetrics`

The brief originally pointed at the existing failure-metrics
slot, but `crates/dynomite/` is on the do-not-touch list for
this branch. The shape we shipped (`parking_lot::Mutex`
behind `AaeMetrics`, `record_*` methods, `snapshot()` ->
sorted entries) mirrors `FailureMetrics` exactly so a
future merge into one shared handle is mechanical.

### Tests

Ten new unit tests in `aae/metrics.rs` (snapshot/load
round-trip, corruption counting, gauge overwrite, dropping
zero-count writes); one new test in `aae/repair.rs`
(`resolve_all_records_dispatched_metric`); two new tests in
`aae/scheduler.rs`
(`install_metrics_routes_observations_to_handle`,
`observations_without_metrics_handle_are_noops`).

## R5: `dyn-admin aae-status`

### Wire format

* New PBC opcodes `DynAaeStatusReq = 220`,
  `DynAaeStatusResp = 221`.
* Two new prost messages: `DynRpbAaePeerStatus` (peer-row
  shape, six fields) and `DynRpbAaeStatusResp` (peers list
  + snapshot metadata + tree shape, eleven top-level
  fields). Round-trip tested in
  `proto/pb/messages.rs::admin_tests::aae_status_resp_round_trips`.

### Server side

* New `aae/status.rs` module. `AaeStatusSnapshot` (the
  Rust-level shape returned by the trait), `AaePeerStatus`,
  `AaeStatusProvider` trait, `NoopAaeStatusProvider`.
* `process_frame` gains an `&dyn AaeStatusProvider`
  parameter; the existing `handle_conn_with_hooks` wires
  the noop default so existing callers compile unchanged.
* New public entry points:
  `serve_pbc_with_aae_status(listener, ds, admin, aae_status)`
  and
  `handle_conn_with_aae_status(stream, ds, admin, hooks, aae_status)`.
  Both delegate to a private `serve_pbc_full` /
  `handle_conn_full` that owns the new param.

### Client side: `crates/dyn-admin/src/commands/aae_status.rs`

* `clap` subcommand `aae-status [--node <host:port>] [--json]`.
* Human renderer: bordered per-peer table plus a block of
  `key: value` lines for the snapshot metadata and tree
  shape.
* JSON renderer: stable schema mirroring `AaeStatusReport`.

### Tests

* Unit: four tests in `commands/aae_status.rs` (human, JSON,
  empty, PB->Rust mapping).
* Integration: three end-to-end tests in
  `crates/dyn-admin/tests/aae_status_integration.rs`. The
  human test drives the CLI binary over a real PBC listener
  with a synthetic provider; the JSON test asserts every
  field round-trips; the third test confirms the
  default-listener path (no provider configured) returns
  an empty snapshot rather than erroring.
* CLI smoke (`tests/cli_smoke.rs`) gains `aae-status` to
  the list of subcommands `--help` is required to mention.
* Server-side: two PBC dispatch tests in
  `crates/dyniak/src/server.rs`
  (`aae_status_default_provider_returns_empty_snapshot`,
  `aae_status_custom_provider_returns_live_snapshot`).

## Default behavior (sanity check)

* `ConfAae::per_token_exchange` defaults `false`. Existing
  deployments that build a sweep plan via
  `SweepPlan::new(...)` see no behavior change.
* New metric counters initialise to zero. `AaeMetrics::new()`
  is unused until an embedding installs it; the engine is
  not driving any counters from this branch.
* `dyn-admin aae-status` is opt-in (operators must invoke
  it). Server-side the default `NoopAaeStatusProvider`
  returns an empty snapshot so a CLI invocation against
  any existing `dynomited` build succeeds with empty
  output rather than failing.
