# 2026-05-30 turbovec integration

Added the `turbovec = "0.8"` crate to `crates/dynvecdb/` and
wired its TurboQuant codec into the engine's encoding and ANN
search layers. This is the workspace's third vector encoding,
joining `Int8Quantized` (4x compression) and `Fp16` (2x).

## What landed

* `Codec::Turbovec2Bit`, `Codec::Turbovec3Bit`,
  `Codec::Turbovec4Bit` enum variants on
  `crate::encoding::Codec`. The variants expose the bit width
  via a new `Codec::turbovec_bits() -> Option<u8>` helper so
  the storage layer can dispatch on codec without sprinkling
  `match` arms across the hot paths.
* `crate::encoding::Turbovec` encoder. `encode` produces an
  `EncodedVector` whose `bytes` field is the source vector's
  `f32` little-endian payload (passthrough). `decode`
  recovers it byte-for-byte. The headline 2/3/4-bit packed
  representation lives in the per-table
  `crate::turbo_index::TurboTable`, not in the row payload;
  see the rationale below.
* `encoding::encode_turbovec(vector, bits) ->
  Result<EncodedVector, EncodingError>` and
  `encoding::decode_turbovec(ev, bits) -> Result<Vec<f32>>`
  helpers that match the spec's surface.
* `encoding::distance_turbovec(query, stored, metric) ->
  f32`. Builds an ephemeral 1-vector
  `turbovec::TurboQuantIndex`, runs the SIMD
  `search_with_mask` against it, and converts the
  inner-product surrogate to the requested `Distance` metric.
  Used for single-pair scoring and parity tests; the bulk
  search path uses the per-table TurboTable directly.
* `crate::turbo_index::TurboTable`. A drop-in alternative to
  `HnswIndex` that holds a `turbovec::TurboQuantIndex` plus a
  slot-to-NodeId map. Selected by
  `crate::storage::AnnContainer::new` when the table's codec
  is `Turbovec*`. L2-normalises queries and stored vectors at
  ingest time when the metric is `Cosine` or `Euclidean` so
  turbovec's inner-product surrogate doubles as a cosine
  estimate.
* `crates/dynvecdb/build.rs`. Probes for `libopenblas` in
  standard locations (Debian / RHEL / Homebrew / Nix store)
  and forwards the search path via
  `cargo:rustc-link-search=native=...`. Also honours
  `OPENBLAS_LIB_DIR` for CI / non-standard installations. See
  "openblas link-time dependency" below.
* Tests at `crates/dynvecdb/tests/turbovec.rs`. Seven
  integration tests covering round-trip, SIMD-vs-brute-force
  distance agreement, recall@10, and the compression-ratio
  formula.
* Bench update: a new `bench_search_turbovec` group at
  `crates/dynvecdb/benches/throughput.rs` mirrors the
  existing 10k-vector top-10 search benchmark with
  `Codec::Turbovec4Bit + Distance::Cosine`.
* Doc update: the encoding section of
  `docs/dynvecdb/architecture.md` now documents all three
  codecs and the trade-off matrix for picking between them.

## Why a passthrough at the row layer

The naive interpretation of the spec ("Turbovec stores
compressed bytes per row") clashes with turbovec 0.8's public
API in two places:

1. `turbovec::TurboQuantIndex` is the only encode entry point.
   It owns a flat packed-codes buffer keyed by positional slot
   plus the per-table TQ+ calibration that is fitted on the
   first `add` and frozen for the lifetime of the index. The
   `packed_codes` and `scales` accessors are `pub(crate)`,
   so per-vector blob extraction would require a tempfile
   round-trip through `write` / `load` per encode call.
2. There is no public `decode` operation. The compressed
   representation is only meaningful inside the SIMD scoring
   kernel; turbovec deliberately ships an encode-only codec
   because reconstruction is lossy by construction (2-4 bits
   per coordinate after the JL-style rotation).

Either constraint on its own would force the integration into
a per-table-index shape; together they make it the only
reasonable choice. The row layer therefore stores the original
`f32` bytes (passthrough), and the per-table `TurboTable`
holds the compressed packed codes that the SIMD kernels
consume. Compression accrues in the in-memory ANN index, not
in the on-disk row, but the *search-time* cost (the metric
the user actually feels) is fully on turbovec's SIMD path.

This is the "thin shim around `quantize` and `search`" fallback
described in the task brief, taken verbatim because the
`encode` / `decode` per-vector contract was not satisfiable
with turbovec's exposed API.

## openblas link-time dependency

`turbovec` -> `ndarray` enables the `blas` feature on Linux
and macOS. This in turn requires `-lopenblas` at link time.
The dynvecdb build does not reference any openblas symbols
directly, but the linker still has to find the library. The
new `build.rs` resolves this by:

1. honouring `OPENBLAS_LIB_DIR` if set;
2. probing `/usr/lib/x86_64-linux-gnu`, `/usr/lib64`,
   `/usr/lib`, the Homebrew prefixes, and `/nix/store/*-openblas-*/lib`;
3. emitting a `cargo:warning=` if nothing is found, so the
   subsequent link error has context.

CI environments without openblas should set
`OPENBLAS_LIB_DIR` once.

## Distance semantics

turbovec's score is an inner-product surrogate that already
factors in the per-vector length renormalisation correction
(the `||v|| / <u, x_hat>` scalar described in the turbovec
README). The TurboTable maps it to dynvecdb's
smaller-is-closer convention as follows:

| Metric      | Pre-add transform | Score mapping             |
|-------------|-------------------|---------------------------|
| DotProduct  | identity          | `score = -similarity`     |
| Cosine      | L2-normalise      | `score = 1 - similarity`  |
| Euclidean   | L2-normalise      | `score = sqrt(max(0, 2 - 2*sim))` |

For `Euclidean`, the surrogate orders identically to L2
distance on the L2-normalised inputs but is not bit-equal to
the true L2 score. Embedders that need exact L2 should use
the HNSW path (default) or the new turbovec path with a
post-rerank against the row-stored f32 bytes.

## Open questions

* The 2-bit and 3-bit recall@10 numbers are not asserted yet;
  only round-trip tests exist. Adding 2-bit / 3-bit recall
  tests is mechanical and can fold into a follow-up.
* The bench `search_turbovec_4bit` group prints throughput
  but does not yet compare-against a baseline. A follow-up
  should commit a baseline under `crates/dynvecdb/benches/`
  alongside the existing one.
* `Distance::Euclidean` with a `Turbovec*` codec uses the
  cosine surrogate; a future change could optionally rerank
  the top-K against the row-stored f32 bytes for embedders
  who care about exact L2 ordering.
