# 2026-05-19 Stage 4: Configuration

Branch: `stage/4-conf`

## Goal

Port `_/dynomite/src/dyn_conf.{c,h}` plus the YAML configuration corpus
under `_/dynomite/conf/` into a typed Rust schema (`dynomite::conf`).
The schema must round-trip every shipped YAML, mirror every
`conf_validate_pool` check from the C reference, and expose
`Config::{parse_str, parse_file, finalize, validate, test_conf}`.

## Design notes

* The YAML structure is a single-key top-level mapping (the pool
  name) whose value is the pool body. `Config` exposes that as
  `pool_name -> ConfPool`. Multiple pools are rejected with
  `ConfError::TooManyPools`.
* All `CONF_UNSET_NUM` / `CONF_UNSET_BOOL` / `CONF_UNSET_HASH` fields
  are `Option<T>`. `apply_defaults` mirrors `conf_validate_pool`
  defaulting and is intentionally idempotent.
* `serde(deny_unknown_fields)` on `ConfPool` produces typed
  `ConfError::UnknownKey` when the YAML contains a directive the C
  parser would reject in `conf_handler`.
* `ConfListen`, `ConfServer`, `ConfDynSeed`, and `TokenList` ship
  custom `Serialize` / `Deserialize` impls so the original raw string
  shapes (e.g. `127.0.0.1:8101`, `host:port:rack:dc:tokens`) survive
  the round-trip while still validating at parse time.
* `validate` is split into `validate_numeric_ranges`,
  `validate_mbuf_size`, `validate_max_msgs`, and the cross-field
  checks (`secure_server_option`, `read/write_consistency`,
  `pem_key_file` requirement, `servers` cardinality, `data_store`
  membership, `hash_tag` length).
* The deprecated directives (`distribution`, `server_connections`)
  are accepted and ignored, matching `conf_set_deprecated`.

## Ambiguities resolved

Both recorded under `docs/parity.md` -> `# Ambiguities`:

1. `mbuf_size` / `max_msgs` defaults remain disabled, matching the
   commented-out code path in `conf_validate_pool`. Pinned by a
   regression test that exercises every reject path.
2. The C parser's "exactly one server" invariant (a
   single-pointer field guarded by an assert) is reproduced as a
   typed `ConfError::BadServer` rather than a panic.
