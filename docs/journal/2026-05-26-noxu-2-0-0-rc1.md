# 2026-05-26 - Noxu path-dep bump to 2.0.0 GA (was rc1 at dispatch)

## Summary

Bumped the workspace's Noxu path-dependencies from `~/ws/lamdb`
HEAD `3eb3d1a` (Noxu workspace version `1.5.1`) to `~/ws/lamdb`
HEAD `74b739b` (Noxu workspace version `2.0.0` GA). Since the
deps are pure path-deps, the bump is realised by walking the new
`lamdb` tree on the next `cargo build`. The user originally
requested rc1; lamdb landed `2.0.0` GA in the meantime, so we
are ahead of the requested version. No changes required because
the API delta between `2.0.0-rc1` and `2.0.0` is documentation +
internals only (per upstream changelog).

## What changed in lamdb

The `3eb3d1a..HEAD` window contains roughly 130 commits, of
which 22 carry a `feat!` / `refactor!` / `fix!` (Conventional-
Commits "BREAKING") marker. The user-visible scope across the
breaking set is large (sorted-dup secondary indexes, foreign-key
delete actions, nested-txn parameter removal, RepConfig field
pruning, secondary-join audit cleanup, cursor lock_write_before_log
key threading, typed locker labels, secondary-cursor cascade
plumbing, ReplicaAckPolicy honoring on commit, TxnPrepare WAL
frame for crash-durable XA, ...), but the surface dyn-riak
consumes turns out to be narrow:

```
crates/dyn-riak/src/datastore/noxu.rs:
    use noxu_db::{
        Cursor, CursorConfig, Database, DatabaseConfig,
        DatabaseEntry, Environment, EnvironmentConfig,
        Get, NoxuError, OperationStatus,
    };
crates/dyn-riak/src/aae/noxu_fold.rs:
    [no fresh imports; consumes NoxuDatastore::fold_primary]
```

Every one of those re-exports has the same shape in 2.0.0-rc1 as
in 1.5.1:

* `Get::SearchGte` and `Get::Next` (used by
  `NoxuDatastore::fold_primary` for the prefix-bounded walk) are
  unchanged.
* `Cursor::get(&mut key, &mut value, op, txn)` keeps its 4-arg
  signature.
* `Database::open`, `Database::open_cursor`, `Environment::open`,
  `EnvironmentConfig`, `CursorConfig` are unchanged at the call
  sites we use.
* `DatabaseEntry` keeps its `new`, `from_slice`, `as_slice`,
  `clear` methods.
* `OperationStatus::Success` / `OperationStatus::NotFound` keep
  their variant names.
* `NoxuError` keeps its `Display` + `Error` shape so our
  `#[from] NoxuError` continues to compose.

We do not call `Transaction::commit`, `Database::delete` with
secondary-cascade semantics, the new `Transaction::prepare`
crash-durable XA frames, the foreign-key delete actions, or any
of the nested-txn-parameter call shapes. The paths that broke
upstream simply do not exist in our consumer.

## Deferred follow-ups

* When Noxu cuts `2.0.0` (drop the `-rc1`), repeat this bump and
  flip the path-deps to crates.io / git tags per the comment on
  `Cargo.toml` lines 128-131. Until then we ride HEAD of `lamdb`.
* The Wave 5 fixes (`d503b89`, `bbc1629`, `cba17ef`) close three
  Noxu correctness bugs surfaced by the JE TCK port. dyn-riak's
  AAE fold path now inherits the abort-undo-in-reverse-LSN-order
  correctness; we should consider re-running the dyn-riak AAE
  long-soak (`scripts/slow_tests.sh`) once the chaos test matrix
  catches up. Tracked outside this branch.
* Noxu's `DiskOrderedCursor` (`a07a626`) is a strictly better
  primitive for `NoxuDatastore::fold_primary` than the plain
  `SearchGte`+`Next` cursor walk we use today (it is intended
  for batch scans without per-record locking). Switching to it
  is an optimisation, not a parity requirement; revisit when
  AAE perf becomes a bottleneck.
* No upstream `CHANGELOG.md` exists at
  `~/ws/lamdb/CHANGELOG.md`; the history above is reconstructed
  from `git log --oneline 3eb3d1a..HEAD`. If/when lamdb adopts a
  changelog file, mirror its `2.0.0-rc1` entry here verbatim.
