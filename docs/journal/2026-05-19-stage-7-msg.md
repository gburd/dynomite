## Summary

Implements `dynomite::msg` (the in-memory request/response shape)
and `dynomite::proto::dnode` (the wire codec for Dynomite peer
traffic). Both modules are added to `crates/dynomite/src/lib.rs`
behind the existing `#![forbid(unsafe_code)]` guard.

The data-shape side of every C function listed in PLAN.md Stage 7
is in place. Connection-coupled paths (recv/send, conn FSM,
timeout queues, peer forwarding) are explicitly deferred to Stage 9
with parity-row notes; gossip-message decoders are deferred to
Stage 10 with parity-row notes.

## Ambiguities

- The on-the-wire `same_dc` field is read by the C parser one
  digit at a time (no decimal accumulator). Multi-digit values
  would only retain the last byte. The Rust parser matches this
  behaviour: a digit is stored, then a space transitions to the
  next state. In practice the field is always 0 or 1.
- The C parser's recovery path on bad bytes flips between
  `MSG_PARSE_AGAIN` and `MSG_PARSE_REPAIR` based on whether the
  bad mbuf is full or partially consumed; that recovery is
  inseparable from the connection's mbuf chain management. The
  Stage 7 Rust parser surfaces the bad-byte case as a typed
  `ParseStep::Error { consumed }` so Stage 9 can implement the
  same recovery on top of the conn FSM.

## Deviations

- The dispatcher `dmsg_process` returns a typed
  `DmsgDispatch::{Bypass, Forward}` rather than a raw `bool`.
  The semantics are identical to the C implementation
  (`true` -> `Bypass`, `false` -> `Forward`).
- `ResponseMgr` boxes its retained `Msg` values to break the
  recursive layout cycle (`Msg` owns `ResponseMgr`,
  `ResponseMgr` owns `Vec<Box<Msg>>`). The C engine sidesteps
  the issue by storing `struct msg *` pointers; the Rust port
  preserves ownership via `Box`. No observable behaviour
  change.
- `is_read_repairs_enabled` is backed by a `OnceLock<bool>`;
  `set_read_repairs_enabled` returns `false` on the second and
  later calls instead of overwriting. The C engine's
  `g_read_repairs_enabled` is set once during config validation
  and never changed afterwards, so the OnceLock semantics are
  appropriate and slightly stricter than the C global.
