# Stage 6 - Crypto

Date: 2026-05-19
Branch: `stage/6-crypto`
Worker: stage-6 sub-agent

## Summary

Implements `dynomite::crypto`: AES-256-CBC payload crypto, RSA wrap
and unwrap, base64 helpers, and PEM key loading. Wires the module
into `crates/dynomite/src/lib.rs` behind the existing
`#![forbid(unsafe_code)]` guard. Tests cover unit-level invariants,
property tests for round-trip stability, and a frozen-output
fixture under `crates/dynomite/tests/fixtures/crypto/`.

## Deviations from the C reference

Recorded in `docs/parity.md` under `## Deviations`. Summary:

1. **Long-lived static `EVP_CIPHER_CTX *` replaced by per-call
   `openssl::symm::Crypter`.** The C globals are not safe under
   concurrent worker tasks; the Rust port allocates a fresh
   crypter per call. Functionally equivalent.

The AES wire format, base64 encoding, and RSA padding all match
the C reference byte-for-byte after the review-response commits
(see "Review response" below).

## Ambiguities

- The Stage 6 brief asserted that the AES output format reproduced
  the C reference exactly, but the layout in `dyn_crypto.c` is more
  surprising than the brief's prose suggested (no IV prefix,
  AES-128, key reused as IV). The review-response commits below
  bring the Rust port into byte-for-byte agreement with the C
  reference.
- The Stage 6 brief described the RSA padding choice as PKCS#1
  v1.5 "to match C" but the C actually uses OAEP. Resolved by
  commit 01ed58f (Stage 6 follow-up).
- The dyn_aes_encrypt brief signature accepts a destination
  `MbufPool` rather than mutating a single caller-supplied
  `Mbuf`. We chose the pool-based form because it is a closer
  fit for the tokio reactor model than the C in-place
  `mbuf->end_extra` reservation.

## Cross-impl test

The fixture (`tests/fixtures/crypto/{aes_key,plaintext,cipher}.bin`)
is regenerated by `examples/gen_crypto_fixture.rs` and pinned by
`tests/stage_06_crypto.rs::wire_format_pin_round_trip`, which
decrypts the committed ciphertext and re-encrypts the plaintext
to confirm byte-for-byte determinism. After the review-response
commits the cipher is AES-128-CBC with the key reused as the IV,
so the output is deterministic and reproducible by any compliant
implementation. The committed `cipher.bin` was independently
verified against the OpenSSL CLI:

```
printf 'dynomite stage 6 crypto fixture: round-trip me' | \
  openssl enc -aes-128-cbc \
    -K  1032547698badcfe0123456789abcdef \
    -iv 1032547698badcfe0123456789abcdef
```

produces the same 48 bytes that `Crypto::aes_encrypt` produces.

## Review response

Date: 2026-05-20.

Review report: `git show review/stage-6:docs/journal/review-stage-6-7c7425e1.md`.
Verdict: REQUEST_CHANGES, blocking on a stale-rustdoc defect and
advisory on three wire-format deviations.

The lead's follow-up dispatch determined that the AES, base64,
and RSA deviations were not blessed and must be reverted: the
port's mandate is to reproduce C behavior, with security
improvements deferred to a post-port hardening pass. This round
of commits brings the Rust port into byte-for-byte agreement with
the C reference for all three wire formats.

Commits applied on top of `01ed58f`:

1. `61ccac9` fix(crypto): match C AES-128-CBC with key reused as
   IV. Switches `Cipher::aes_256_cbc()` -> `Cipher::aes_128_cbc()`,
   removes the random-IV-prefix encoding, and uses the first 16
   bytes of the 32-byte key buffer as both the key and the IV
   (mirroring `EVP_EncryptInit_ex(ctx, cipher, NULL, key, key)`
   in `dyn_crypto.c:277`). Adds `known_vector_pin` test against
   bytes captured from `openssl enc -aes-128-cbc`. Regenerates
   `tests/fixtures/crypto/cipher.bin` from the new algorithm; the
   bytes are independently produced by the OpenSSL CLI.
   Renames `cross_version_cipher_round_trip` to
   `wire_format_pin_round_trip`. Also scrubs the three stale
   PKCS#1 v1.5 references in `mod.rs` rustdoc that the OAEP
   correction commit (`01ed58f`) had missed (the documentation
   defect that triggered REQUEST_CHANGES).
2. `e5aae2d` fix(crypto): emit padded base64 to match C output.
   Switches `base64_encode` from `STANDARD_NO_PAD` to `STANDARD`.
   The C `BIO_f_base64()` + `BIO_FLAGS_BASE64_NO_NL` path emits
   padded output; the flag suppresses newlines, not padding.
   `base64_decode` continues to accept both forms.
3. (this commit) docs(crypto): remove resolved deviations and
   record review response. Drops the AES and base64 deviations
   from `docs/parity.md` (they no longer describe actual
   deviations) and updates the parity table rows. Updates this
   journal in place to reflect the corrected design rather than
   the original deviation-laden one.
