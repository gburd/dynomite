## Findings

Listed worst first.

### F1. Three conformance tests fail with `redis-server` on PATH (pre-existing since v0.1.0)

* `dynomited::conformance::single_node::quit_closes_socket`
  -- QUIT command times out (5s) instead of getting `+OK\r\n`
  or socket close.
* `dynomited::conformance::three_node_single_dc::each_node_accepts_workload`
  -- SET on a non-local-token node times out.
* `dynomited::conformance::multi_dc::dc_quorum_workload` --
  same shape, multi-DC.

CI did not catch these because the conformance test crate is
gated behind `feature = "integration"` (see `tests/conformance.rs:1`)
and the default `cargo nextest run --workspace` does not
enable that feature. CI runs the conformance gate in
`scripts/check.sh` only when both `cargo-nextest` AND
`redis-server` are on PATH; CI hosts didn't have redis,
so the tests silently skipped at the binary level.

Confirmed pre-existing: `git checkout v0.1.0` reproduces all
three failures.

**Decision needed**: do we (a) fix the underlying behaviour
(QUIT timeout suggests a real bug in the QUIT plumbing under
the conformance harness's client; the 3-node SET timeouts
suggest the dispatcher routes to a peer that doesn't reply
in this single-host topology), or (b) mark them `#[ignore]`'d
with an issue link until gossip wiring steps 3-5 land?

### F2. CI gate `cargo fmt --all -- --check` was broken

Adding the noxu path-deps in `feat/riak-compat-plan` made
`cargo fmt --all` walk the lamdb workspace and report 3175
diffs (none in our code). **Fixed in this commit**:
`scripts/check.sh` now uses `cargo fmt -p dynomite -p
dynomited -p dyn-hash-tool` so format checks stay scoped to
our crates.

### F3. CI gate `cargo clippy --all-features` was broken

A test helper `read_until` in
`crates/dynomited/tests/integration.rs:90` was added by
`3b6c00b` but never used. Compiles fine in default features
(the file is `#![cfg(feature = "integration")]`-gated) but
fires `dead_code` under `--all-features`. **Fixed in this
commit**: removed the helper.

### F4. CI gate `scripts/check_ascii.sh` was broken

Two docs created this session smuggled em-dashes (U+2014):

* `docs/riak-comparison.md` (11 instances)
* `docs/upstream-fork-review.md` (9 instances)

**Fixed in this commit**: replaced em-dashes with ` -- `.
This was an `sed` against the raw bytes; spot-check confirmed
no other characters affected.

### F5. CI gate `scripts/check_no_port_comments.sh` was broken

Two docs created this session contained the phrases AGENTS.md
section 1.3 forbids outside README/NOTICE/LICENSE:

* `docs/book/src/operations/metrics.md:11` -- "scripts that
  target the C reference engine continue to work" --
  rephrased to "scripts that target the legacy schema".
* `docs/riak-comparison.md:4` -- "our Rust port of Netflix
  Dynomite" -- rephrased to "this Rust implementation".

**Fixed in this commit**.

### F6. AGENTS.md and README.md reference `_/dynomite/` and `test/`, both removed

* `_/dynomite/` was the vendored Netflix C source tree; it
  was removed in commit `2561d13` (2026-05-21).
* `test/` was supposed to hold a Python conformance harness;
  it never existed in this checkout (the scenarios were
  re-implemented in Rust as
  `crates/dynomited/tests/conformance/`).

Both are still listed in the AGENTS.md section 2 layout map
and referenced in section 1, section 3 (`DYNOMITE_C_REF` env
var), section 14, and section 16. README.md says "carried
under `_/dynomite/` for reference".

`crates/dynomited/tests/differential.rs` falls back to
`workspace_root().join("_/dynomite/src/dynomite")` when
`CONFORMANCE_C_BINARY` is unset, so the rig has been
silently skipping ever since.

**Fixed in this commit**: AGENTS.md, README.md, and
`differential.rs` updated to acknowledge that the C tree is
no longer vendored. The differential test now requires
`CONFORMANCE_C_BINARY` explicitly and prints a skip notice
otherwise.

### F7. `--log-format` is not in `--help`

`dynomited --help` reproduces `dn_show_usage` from the C
reference verbatim, so it does not list any new flags.
`--log-format` is parsed by clap and works correctly; it
just does not appear in the usage string.

**Not fixed**: cosmetic-only. Either we accept the
discrepancy (the C-reproduced help is a parity goal) or we
add a "see `man 1 dynomited`" line. Logged for follow-up.

### F8. Phi-accrual is plumbed but never called

`Peer.fd: PhiAccrual` exists on the struct, has `default()`
construction in `Peer::new`, and has `failure_detector() /
failure_detector_mut()` accessors. **Zero callsites** outside
unit tests. The detector is operationally inert.

This matches the session note that gossip-wiring steps 3-5
are deferred. Honest description: "scaffolding present, not
yet active". Logged in `docs/post-chaos-queue.md` as the
gossip-wiring work item.

### F9. Three load-correlated nextest flakes

Under `--all-features` (which doubles the test count to 708
and increases parallel pressure), three tests sometimes fail
when scheduled together but always pass in isolation:

* `dynomited::pidfile::tests::flock_retry_succeeds_when_holder_drops_during_window`
* `dynomite::crypto::aes::tests::wrong_key_fails_padding_check`
* `dynomited::conformance::multi_dc::dc_safe_quorum_workload`

The flock test is a known load-correlated flake (sister to
the `pidfile_is_written_and_removed` test we `#[ignore]`'d
at v0.1.0).

The crypto test failing under load is **suspicious**; the
test is pure compute against fixed keys, so a load-induced
failure suggests test-fixture state sharing or an OS-level
syscall timing dependency we have not pinned down.

**Not fixed**: needs investigation under `RUST_TEST_THREADS=1`
to isolate. Not blocking the audit.

### F10. End-to-end OTLP test is `#[ignore]`'d

`otlp_grpc_bytes_reach_mock_listener` actually works (3.0s,
verified manually). The `#[ignore]` is to keep CI fast. We
should consider running it on a slow-tests CI lane so the
"OTLP exporter actually puts bytes on the wire" claim has
continuous coverage.

**Not fixed**: deliberate trade-off; logged for slow-lane.

## What this audit does NOT verify

* `cargo deny check` and `cargo audit`: skipped (advisories
  list rotates daily; not material to "does the code do what
  it claims").
* `mdbook build`: requires `mdbook` on PATH (only available in
  `nix develop`).
* `scripts/quickfuzz.sh 60`: each fuzz target is ~60s, plus
  build time.
* `scripts/coverage_gate.sh`: takes ~5min for the cargo-llvm-cov
  full pass.
* The 1h Stage-16 chaos test is a pre-tag gate, not a per-audit
  gate.

These are the four genuinely slow gates; they pass on `main`
when the rest does, and we have no signal that anything in
this session changed their result.

## Bottom line

The code does what it advertises, with three caveats:

1. The conformance suite has had three legitimate failures
   since v0.1.0 that no one was running because CI hosts
   didn't have `redis-server`. (F1)
2. Three CI hygiene gates were silently broken on `main` for
   docs created this session. (F2--F5; **all fixed**.)
3. AGENTS.md and README.md describe a tree layout that no
   longer matches reality. (F6; **fixed**.)

After this commit, the project's stated CI gate
(`scripts/check.sh`) is consistent with what `main` actually
does, and a future agent reading AGENTS.md will not be sent
chasing files that no longer exist.

The recommended next step is F1: decide whether the three
pre-v0.1.0 conformance failures get a real fix (likely a
half-day) or `#[ignore]`'d with a tracking issue. Without
that, every future "does conformance pass?" question gets
a misleading answer.
