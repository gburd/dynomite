## Findings

Listed worst first.

### F1. Three conformance tests fail with `redis-server` on PATH (pre-existing since v0.1.0)

* `dynomited::conformance::single_node::quit_closes_socket`
  -- QUIT command times out (5s) instead of getting `+OK\r\n`
  or socket close.
* `dynomited::conformance::three_node_single_dc::each_node_accepts_workload`
  -- SET on a non-local-token node times out.
* `dynomited::conformance::multi_dc::dc_quorum_workload` --
  same shape, multi-DC.

CI did not catch these because the conformance test crate is
gated behind `feature = "integration"` (see `tests/conformance.rs:1`)
and the default `cargo nextest run --workspace` does not
enable that feature. CI runs the conformance gate in
`scripts/check.sh` only when both `cargo-nextest` AND
`redis-server` are on PATH; CI hosts didn't have redis,
so the tests silently skipped at the binary level.

Confirmed pre-existing: `git checkout v0.1.0` reproduces all
three failures.

**Decision needed**: do we (a) fix the underlying behaviour
(QUIT timeout suggests a real bug in the QUIT plumbing under
the conformance harness's client; the 3-node SET timeouts
suggest the dispatcher routes to a peer that doesn't reply
in this single-host topology), or (b) mark them `#[ignore]`'d
with an issue link until gossip wiring steps 3-5 land?

### F1 resolved

Commit `490a258 fix(cluster): three pre-v0.1.0 conformance
failures (F1 from audit)`. Three independent bugs:

1. **QUIT deadlock** in `crates/dynomite/src/net/client.rs`.
   The placeholder pushed onto `omsg_q` for QUIT was never
   popped because the dispatcher returned `Drop` (no
   response generated). The outer loop's exit condition was
   never satisfied. Fix: synthesize the `+OK\r\n` reply Redis
   would have sent and route it through the same response
   path, which pops the placeholder cleanly. Adds
   `dynomite::msg::response::make_simple_redis`.

2. **Asymmetric dnode framing** in
   `crates/dynomite/src/net/dnode_client.rs`. The peer-receive
   side wrote raw response payload back to the originator
   without prepending a dnode header. The originator parses
   incoming bytes through `DnodeParser` which requires a
   header. Mismatch caused the originator parser to sit in
   `NeedMore` indefinitely and the originating client to time
   out. Fix: prepend a `DmsgType::Res` header to every
   response.

3. **Peer state never transitions out of `Down` without
   gossip** in `crates/dynomited/src/server.rs`. Remote peers
   built by `seed_to_peer` initialise to `PeerState::Down`.
   The dispatcher's `collect_routable` filters on
   `is_routable()`, so every cross-node route resolved to
   `NoTargets`. Fix: have `peer_supervisor` publish peer-state
   transitions tied to the local TCP link (Normal on connect,
   Down on disconnect). When the gossip runtime lands it can
   authoritatively override.

Diagnosis credit: a focused subagent (id `9ab6b5ce`) walked
the dispatcher path under nextest with a temporary
instrumentation harness, confirmed the Down-state filter, and
implemented the supervisor-driven state-publish fix. Diagnostic
instrumentation reverted in the same diff.

Post-fix: conformance suite is **34/34** under
`--features integration --profile conformance` (was 31/34
since v0.1.0). Default `cargo nextest run --workspace`:
**650/650** unchanged. Doctests: 11 pass (added one for
`make_simple_redis`). Clippy (default + `--all-features`):
clean. Fmt scoped to our crates: clean. All three hygiene

### Latent bug surfaced

`make_error` produces a metadata-only Msg (no mbuf payload).
Any code path that returns `DispatchOutcome::Error` -
`NoTargets`, peer-channel full, datastore-channel full -
sends 0 bytes and the client hangs until timeout. Tracked as
a half-day item in `docs/post-chaos-queue.md`. Until that is
fixed, every operational error masquerades as a hang.

### Bug #12 resolved (chaos-injector)

Commit `10df1c8 fix(chaos): kill any stale dynomited before
restart (bug #12)`. Pass-2 surfaced a runaway-restart loop on
arnold (510 events). Root cause: when start-host.sh crashed
mid-flight (between the binary spawn and the pidfile write),
the recovery branch saw a missing pidfile-tracked process AND
another untracked dynomited still bound to the listen port.
start-host.sh would then fail with "Address already in use"
on every retry.

Fix: add `kill_stale_dynomited()` to chaos-injector.sh.
pgrep for any dynomited whose argv contains our DC's config
path, SIGKILL all matches, wait for reap, drop the stale
pidfile. Called from `restart_dynomited` so every restart
starts from a clean slate.

The pgrep filter scopes to `dynomited.*$RUN/dynomite.yml` so a
neighbour DC running on the same host (per-DC `$RUN` path) is
unaffected.

shellcheck clean. `bash -n` clean.

* **F9** (three nextest flakes under `--all-features`):
  **resolved 2026-05-24**.
  * `dynomited::pidfile::tests::flock_retry_succeeds_when_holder_drops_during_window`:
    the holder thread held the flock for 80ms while the
    contender's retry budget was 20 attempts x 10ms = 200ms.
    Under load the holder's `std::thread::sleep` and/or the
    contender's per-attempt sleeps both jittered; when the
    contender ran out of retries before the holder released,
    the test surfaced `WouldBlock`. Test rewritten to use a
    500-attempt budget (~5 seconds total), a 30ms holder hold,
    and an `mpsc` ack channel so the assertion can confirm we
    actually retried at least once. Since the test owns its
    own `tempfile::tempdir()` it has no cross-test contention,
    so the per-test temp directory is sufficient isolation.
  * `dynomite::crypto::aes::tests::wrong_key_fails_padding_check`:
    statistical, not load-correlated. Decrypting a single
    16-byte ciphertext block with a random wrong key produces
    a uniformly random 16-byte block; PKCS#7 padding
    accidentally validates with probability ~1/255 per attempt
    (the last byte must be a value `n` in 1..=16 followed by
    `n-1` matching bytes). The original test was a single
    trial and false-passed about 0.4% of the time, which is
    why the user observed it more often under load (more
    samples per CI run). Test rewritten to iterate up to 32
    independently-generated key pairs and assert at least one
    surfaces the padding-rejection path. Bound on false
    failure: `(1/255)^32` < 1e-77.
  * `dynomited::conformance::multi_dc::dc_safe_quorum_workload`:
    load-correlated multi-process test. Root cause was that
    `cargo nextest run --workspace --all-features` ran the
    `conformance`, `differential`, and `dynomited::integration`
    binaries under the default profile (parallel), where they
    raced on ephemeral-port allocation. Resolved by adding a
    `default-filter = "not (binary(conformance) |
    binary(differential) | binary(integration))"` clause to
    `.config/nextest.toml`'s default profile and updating
    `scripts/check.sh` so the conformance pass under
    `--profile conformance` (test-threads=1) explicitly
    includes `--test integration` in addition to the existing
    `--test conformance --test differential` set. The default
    workspace gate still observes 728 tests (>= 727 floor);
    `--all-features` under the default profile observes 733
    tests, the conformance profile picks up the remaining 54.
    Twelve consecutive `cargo nextest run --workspace
    --all-features --no-fail-fast` invocations all passed
    cleanly post-fix; the earlier flake rate was 1-2 per 8
    runs. (A separate `dynomite::read_repair::dc_quorum_get_
    returns_majority_value_and_repairs_divergent_replica`
    flake surfaced once during the stress run and is unrelated
    to the F9 list; logged for follow-up.)
