## Bugs found and their disposition

### Bug A: backend_supervisor reconnect storm (1781% CPU)

**Symptom**: dynomited hit a tight reconnect loop against a
backend that returned protocol-parse errors on every probe;
caught at one point at 1781% CPU on meh.

**Root cause**: flat 50ms sleep between reconnects with no
backoff, no jitter, no failure budget; only reset on connect-
success rather than first-frame-parse-success.

**Fix**: jittered exponential backoff (50ms-5s, factor 2.0,
jitter [0.5, 1.5]); reset only on a successful frame parse;
new `backend_reconnect_total` Prometheus counter; WARN log
throttled to first 3 + every 10th thereafter.

Landed: commit `c579bbf` (`Merge branch 'fix/busy-loop-cpu'`).

### Bug B: chaos coordinator heredoc malformed start-args

**Symptom**: chaos-injector died at line 310 with
`SEEDS: unbound variable`, leaving the host's dynomited
unsupervised for the entire workload window.

**Root cause**: two-level heredoc + unquoted multi-line
assignment; SEEDS got just the first seed line, subsequent
lines were interpreted as commands starting with `-`.

**Fix**: double-quoted lazy command substitution that
evaluates at source-time on the chaos-injector and preserves
newlines.

Landed: in the same `c579bbf` merge.

### Bug C: start-host backend swap on MODE transition

**Symptom**: when a coordinator's teardown SSH times out (a
remote host wedged), the backend container on `:17100` is
not killed. The next coordinator (different MODE) starts
dynomited configured for the new MODE but the leftover
container is still bound, so dynomited talks the wrong
protocol to the wrong backend. Pass-6 floki had a redis
container leftover from redis mode that survived through
memcache and riak modes; floki contributed 0 successful ops
in memcache and 78% in riak.

**Root cause**: `scripts/chaos-multi-host/start-host.sh` uses
"is :17100 already bound" as a proxy for "the backend is
already up correctly". It should instead verify both
"something is bound" AND "it speaks the right protocol".

**Fix**: NOT YET LANDED. Recommended approach:
1. In start-host.sh, check for existing dyn-chaos-redis-* /
   dyn-chaos-memcached-* containers via the container tool.
2. If a container exists for a different mode, kill it before
   starting the new one.
3. If a native binary is bound to 17100, kill it.
4. Add a positive verification step: probe :17100 with the
   protocol expected by the current MODE and only proceed
   if the probe returns the expected reply shape.

Estimated effort: 1-2 hours.

### Bug D (operational): nuc unreachable via direct SSH

**Symptom**: nuc went unreachable via direct SSH mid-pass-6
and stayed unreachable. ProxyJump via arnold still works.
Coordinator's NUC_DIRECT_SSH path doesn't fall through to
the ProxyJump path.

**Root cause**: the direct SSH path is
`ssh -o ConnectTimeout=5 nuc` which fails because nuc's
Tailscale tunnel to floki appears to be down. ProxyJump
goes nuc-via-arnold-LAN which is healthy.

**Fix**: NOT YET LANDED. Coordinator should fall back to
ProxyJump when direct SSH fails. Already partially in place
for teardown (ProxyJump fallback in dc-nuc-teardown.log
patterns) but not for start.

## What is operationally working

- Heredoc fix B works: arnold + meh + floki all sourced
  start-args correctly in modes where the coordinator could
  reach them. The "0 ok in 2 hours" pattern from pass-5 is gone.
- Retry-backoff works: every host with >0 ok had retry_sleep_ms
  in the 30+ minutes range across the 2-hour window.
- Noxu v2.2.0 path-dep update works: 1567 nextest workspace
  pass under --all-features. dyniak's 431 tests + dynomited's
  82 (--features riak) tests are clean.
- The new gen-fsm, sup, sidejob, throttle, capability, APL,
  events-manager, hashtree, and loom-tests crates are all
  workspace-clean.

## What blocks pass-7

1. **Bug C fix** (start-host backend swap). Without this,
   floki keeps biting on every mode transition.
2. **Bug D fix** (NUC_DIRECT_SSH falls back to ProxyJump).
   Without this, nuc participates in only the first mode
   per pass.
3. **Watchdog** for runaway-CPU dynomited (separate item the
   bug-hunt subagent flagged): if dynomited's CPU stays
   above 800% for 5s, capture stack and dump. Bug A's fix
   is a band-aid; the watchdog catches future bugs of the
   same shape.
