## Ambiguities

None encountered; the C `mbuf` semantics are unambiguous in the
header and source.

The C `mbuf_split` callback (`func_mbuf_copy_t cb`) is used by exactly
one in-tree caller in `dyn_dnode_msg.c` to prepend a header to the
new buffer. The Rust port drops the callback and lets the caller
operate on the returned `Mbuf` directly because the parser stages
have not arrived yet; the parity table records this as a deviation.

The C `mbuf` data-area cursor `end_extra` and the `MBUF_HSIZE` /
`MBUF_MAGIC` constants exist only because the C struct is embedded in
the buffer tail. The Rust port stores cursors in a separate struct,
so `MBUF_HSIZE` / `MBUF_MAGIC` are recorded as omissions and the
`end_extra` boundary is folded into `chunk_size` (the buffer length).

## Deviations recorded in `docs/parity.md`

* `CBuf::push` returns `Err(item)` on full instead of overwriting
  silently like `CBUF_Push`. Callers porting `CBUF_Push` invocations
  must explicitly handle the `Result`. The drop-on-full Stage 1 path
  through `RingChannels` plus `try_send` remains available for the
  gossip ring traffic that needs that semantic.
* `Mbuf::split_off` takes a byte offset relative to the read cursor
  instead of a raw pointer, and the C `cb`/`cbarg` precopy callback
  is dropped (callers prepend bytes to the returned `Mbuf` directly).

## New dependency: `crossbeam-queue`

Added at version `0.3` (workspace pin). Rationale:

* The deliverables ask for `crossbeam_queue::ArrayQueue<T>` as the
  storage primitive for the typed `CBuf<T>`.
* `crossbeam-channel` (already in the workspace) does not expose a
  fixed-capacity owned-slot ring without an `mpsc` boundary; the C
  `CBUF_*` macros are owned-slot rings.
* `crossbeam-queue` is already in the local cargo cache and pulls in
  no transitive deps that are not already present
  (`crossbeam-utils`).
* The crate is `safe`-only, so the workspace `forbid(unsafe_code)`
  invariant remains intact.

## Open questions / follow-ups

* None blocking. The Stage 7/8 message stack will likely want a
  `mbuf_insert_after` shim; the parity row notes that one
  `VecDeque::insert` away. Stage 13 entropy will exercise the
  `MBUF_FLAG_JUST_DECRYPTED` flag bit; the Stage 2 implementation is
  agnostic to which flags are flipped.
* The PLAN-mentioned 100k-operation differential trace against the
  C `mbuf` reference is not part of this stage's exit gate per the
  PLAN's "exit gate" line ("mbuf semantics match the C contract on a
  100k-operation trace; reactor unit test loops a TCP echo
  end-to-end"). The proptest `split_off_then_append_reconstructs`
  covers the chain-correctness property and the reactor TCP echo
  is in place; the differential trace harness is deferred until
  the C build wrapper lands (Stage 14 differential rig).
