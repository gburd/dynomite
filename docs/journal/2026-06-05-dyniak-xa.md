# 2026-06-05 -- dyniak multi-key transactions (XA)

Stage branch: `stage/dyniak-xa`

## Summary

Finished the multi-key transaction work a previous session left
half-landed (the build was broken on three unresolved re-exports in
`crates/dyniak/src/lib.rs`). The crate now exposes an atomic multi-key
transaction surface end to end:

* **Layer 1 (single-environment, complete).** All keys in a batch route
  to one node's Noxu environment and commit in a single engine
  transaction. The storage primitive is
  `NoxuDatastore::transaction(|tx| ...)` (closure API, commit on `Ok`,
  rollback on `Err`) and the wire-shaped `TransactionalStore::execute_batch`.
  Exposed over HTTP at `POST /transactions` and
  `POST /buckets/<bucket>/transactions`.
* **Layer 2 (cross-node 2PC, local-only realisation).** A single-process
  `XaCoordinator` + `XaParticipant` pair drives X/Open XA two-phase
  commit across several local `noxu::xa::XaEnvironment`s, exercising the
  real `xa_start -> work -> xa_end -> xa_prepare -> xa_commit/xa_rollback`
  handshake. The cross-node dnode transport is deferred (see boundary
  below).

## Build fix

`lib.rs` re-exported three symbols the prior session never created:

* `crate::server::handle_conn_with_txn`, `crate::server::serve_pbc_with_txn`
  (PBC extension -- deferred, see follow-ups),
* `crate::proto::http::serve_http_with_txn` (separate HTTP entry point).

Per the brief I wired the transaction handling into the existing HTTP
server rather than standing up a parallel entry point, and dropped all
three re-exports. The HTTP gateway gained a `Route::Transaction`
(`POST /transactions` and `POST /buckets/<bucket>/transactions`); the
handler parses `HttpTxnRequest`, lowers it to a `TxnBatch`, runs it
through the backend's `TransactionalStore`, and renders an
`HttpTxnResponse` (200 committed / 409 aborted / 501 when the backend is
not transactional / 400 for empty or bucket-mismatched batches).

The HTTP layer reaches the transactional backend through a new
`Datastore::as_any` downcast hook (default `None`, overridden by
`NoxuDatastore` to return `Some(self)`). This keeps `serve_http`'s
signature stable -- the transaction route self-detects whether the
configured datastore can transact and returns 501 otherwise -- and
avoids naming the feature-gated `NoxuDatastore` on the dynomite trait
surface.

## Layer 2 boundary and the multi-node design

What runs today is genuine XA two-phase commit, but with every branch
hosted by a local environment in one process. That proves the engine's
XA resource-manager API is integrated correctly (prepare votes,
read-only optimisation, commit and rollback are all driven through
`noxu_xa`), without yet taking on the distributed transport.

The remaining work to make it cross-node:

1. **Branch routing.** Replace the test's `route(&TxnOp) -> usize` with
   the cluster ring: each op's `(bucket, key)` hashes to a primary peer
   (reuse `crate::router::BucketRouter`). Ops whose primaries differ
   become distinct XA branches.
2. **Remote participant.** Add a `dnode`-framed participant that speaks
   the XA verbs (`xa_start`/`end`/`prepare`/`commit`/`rollback`) over the
   peer connection instead of calling a local `XaEnvironment`. The
   coordinator stays protocol-agnostic over a `trait XaBranch`; the local
   `XaParticipant` and the remote stub are two impls.
3. **Coordinator durability + recovery.** Persist the coordinator's
   decision (the gtid and the per-branch vote/commit state) before
   sending phase-2, and on restart use `xa_recover` on each participant
   to resolve in-doubt branches. `XaEnvironment::with_prepared_log` is
   already enabled, so prepared branches survive a participant restart.
4. **Timeouts / heuristics.** A participant that loses the coordinator
   must not hold locks forever; add a presumed-abort timeout and surface
   heuristic completions.

None of this is stubbed in the tree -- the local coordinator is fully
functional and tested; the cross-node path is simply not built yet.

## PBC extension (follow-up)

The PBC `DynRpbTxn*` extension (a dyniak-specific message code outside
Riak's standard 0-83 range carrying a batched-op payload) is deferred.
HTTP was the priority and is done. When it lands it should reuse
`TxnBatch` / `TxnOutcome` exactly as the HTTP path does, with
`handle_conn_with_txn` / `serve_pbc_with_txn` as the server entry points
the prior session had pencilled in. Until then those names do not exist
(re-exports removed) so the crate builds clean.

## dyniak-bench

Added a `txn` op to the Riak HTTP driver (feature `http`): it packs
`DEFAULT_TXN_BATCH_SIZE` puts into one batch and aborts a
`DEFAULT_TXN_ABORT_FRACTION` fraction of them, POSTing to
`/buckets/<bucket>/transactions`. An aborted batch's `409` and a
committed batch's `200` both count as successful ops; anything else is
an error. The body-building logic lives in a transport-free
`txn_workload` module so its smoke tests run under every feature set.
Sample config: `crates/dyniak-bench/examples/riak-txn.toml`.
