## What landed

Two of the five Section-6 follow-ups from
`docs/design/aae-comparison.md`:

* **R1** -- snapshot persistence so the AAE tree survives a
  process restart without a full datastore rescan.
* **R2** -- a native-fold rebuild path for `NoxuDatastore`
  that walks the underlying B-tree once in storage order and
  feeds every primary record into `Tree::insert`.

The shipped wire format and the existing exchange protocol
are untouched; both deliverables are local to the
`crates/dyniak/src/aae/` module plus a small fold helper on
`crates/dyniak/src/datastore/noxu.rs`.

## R1: snapshot persistence

### Module: `crates/dyniak/src/aae/persist.rs`

* `pub const SNAPSHOT_MAGIC: u32 = 0x4141_4531;` ("AAE1")
* `pub const SNAPSHOT_VERSION: u32 = 1;`
* `pub const MAX_FIELD_LEN: u32 = 16 * 1024 * 1024;` -- caps a
  length-prefixed frame so a malformed file cannot trigger a
  multi-GB allocation.
* `Tree::save_snapshot(&self, path: &Path)` -- writes to
  `<path>.tmp` and atomic-renames to `<path>` so a crash
  mid-write cannot leave a torn file. Parent directories
  are created on demand with mode `0700` on Unix.
* `Tree::load_snapshot(path: &Path)` -- reverse operation.
  Errors:
  * `PersistError::Io(io::Error)` -- `NotFound` is the
    common case the scheduler treats as "rebuild".
  * `PersistError::VersionSkew { file, expected }` -- the
    recorded version field does not match
    `SNAPSHOT_VERSION`. Surfaced separately from
    `Corrupted` so operators upgrading across a
    snapshot-format-bump see "old version, rebuild" rather
    than "corrupt, panic".
  * `PersistError::Corrupted(String)` -- magic, body
    layout, or trailer fail to validate. The string
    carries the concrete reason.
  * `PersistError::BadShape(String)` -- the recorded
    `n_time_buckets` / `n_segments` / `time_window_seconds`
    fail their cross-validation invariants.

### File format (v1)

Single binary file, big-endian throughout:

```text
Header (24 bytes):
  4 bytes   magic              0x41 0x41 0x45 0x31  ("AAE1")
  4 bytes   version            u32 = 1
  4 bytes   n_time_buckets     u32
  4 bytes   n_segments         u32
  8 bytes   time_window_secs   u64

Roots:
  4 bytes   roots_count        u32  (must equal n_time_buckets)
  for each: 4 bytes time_bucket u32, 8 bytes root_hash u64

Segments (only non-empty (tb, seg) pairs are recorded):
  4 bytes   segments_count     u32
  for each:
    4 bytes time_bucket        u32
    4 bytes segment            u32
    8 bytes segment_hash       u64
    4 bytes keys_count         u32
    for each key entry:
      4 bytes bucket_len       u32
      bucket_len bytes         bucket bytes
      4 bytes key_len          u32
      key_len bytes            key bytes
      4 bytes vclock_len       u32
      vclock_len bytes         vclock bytes

Trailer (4 bytes):
  4 bytes   magic              0x41 0x41 0x45 0x31  ("AAE1")
```

Decoder cross-checks: recomputed root XOR (from segment
hashes) is compared against the recorded root vector;
mismatch returns `Corrupted`. Empty segments (those with no
directory entries) are not persisted; on load they remain at
the default zero-hash state because XOR over an empty set is
zero.

### Config: `ConfAae` extensions

Two new fields (default-validated):

* `snapshot_interval_seconds: u64 = 300` -- 5-minute
  cadence at which the driver writes a fresh snapshot.
* `aae_state_dir: PathBuf = "/var/lib/dynomite/aae"` --
  directory holding the snapshot. `ConfAae::snapshot_path()`
  joins this with the file name `tree.snapshot`.

`ConfAae::validate()` rejects `snapshot_interval_seconds == 0`.

### Scheduler: `Scheduler::snapshot_due()` / `mark_snapshot_taken()`

Two cadence helpers parallel the existing `poll`/`tick`
surface. The driver's outer loop calls `snapshot_due()`;
on `true`, it invokes `Tree::save_snapshot` and on success
calls `mark_snapshot_taken()` to reset the cadence. A
transient save failure leaves `snapshot_due()` returning
`true` so the next poll retries.

## R2: native-fold path for NoxuDatastore

### Module: `crates/dyniak/src/aae/noxu_fold.rs`

Gated behind the existing `noxu` cargo feature. The new
public method:

```rust
impl Tree {
    pub fn build_from_noxu_fold(
        &mut self,
        db: &NoxuDatastore,
    ) -> Result<(), NoxuFoldError>;
}
```

walks the storage cursor exactly once and feeds every
`(bucket, key, value)` triple into `Tree::insert(bucket,
key, value, 0)`. The win versus a public-API rebuild:

* one B-tree pass, sequential; no per-key MVCC handshake.
* prefix-filtered to skip 2i records at the cursor level
  (they share the same database; their tag bytes are
  `I\0` and `R\0` while primary records use `K\0`).
* bounded memory: no intermediate per-key buffer.

### Helper: `NoxuDatastore::fold_primary`

Public method on the datastore (not behind the feature; it is
already inside the `#[cfg(feature = "noxu")]`-gated module):

```rust
impl NoxuDatastore {
    pub fn fold_primary<F>(&self, f: F) -> Result<(), NoxuDatastoreError>
    where
        F: FnMut(&[u8], &[u8], &[u8]) -> Result<(), NoxuDatastoreError>;
}
```

### Noxu API observation

The brief asked us to look for a `fold(start_key, end_key,
callback)` primitive. Noxu does not expose one. The closest
public surface is `Database::open_cursor` plus the `Get` modes
in `noxu_db::Get` (`First`, `Next`, `SearchGte`, `SearchKey`,
etc.).

The crate-private `scan_prefix` helper in
`crates/dyniak/src/datastore/noxu.rs` already wraps the
cursor into a prefix-bounded walk seeded with `Get::SearchGte`.
That path works for short-range 2i queries but trips a
debug-only panic in `noxu_tree::tree::compress_key`
("key does not start with current prefix") when seeded at the
2-byte `K\0` prefix on a many-key tree -- the prefix-
compression assertion fires inside `cursor_impl::find_range_entry`.

Workaround: a parallel private helper `full_scan` opens the
cursor with `Get::First` and steps forward with `Get::Next`,
filtering the per-record callback by prefix instead of seeding
the seek. `fold_primary` uses `full_scan`. The 2i path remains
on `scan_prefix` because the longer prefixes (with bucket and
index_name embedded) do not trigger the assertion in any of
our currently-shipped tests. If a future fold ever needs
range-bounded semantics, the obvious next step is either a
wider prefix on the seek key or an upstream fix in
`noxu_tree::tree::compress_key`. Recorded as a deferred item.

### vclock and timestamp on the Noxu fold

The Noxu primary value is the opaque object body the client
passed in. `NoxuDatastore` does not split `(value, vclock)`;
that mapping is the caller's job. The fold passes the raw
value bytes into `Tree::insert`'s `vclock` slot. Hash
collisions reduce to "two distinct values must produce two
distinct 64-bit FNV hashes" -- the same collision floor every
content-hashed AAE tree accepts.

Storage records also do not carry a per-key timestamp. The
fold passes `0` for the timestamp argument, routing every
key into time bucket 0. This is acceptable for a cold rebuild
because:

1. The next live observation (a write or sweep tick) will
   re-anchor the key against the wall-clock timestamp of
   that observation through the XOR identity.
2. Bucket 0's segment hashes still produce the right XOR
   aggregate for the rebuild's purposes.

A future Riak-shaped object schema slice that carries a
vclock blob and/or a write-timestamp on every primary record
could feed those values verbatim; the fold would not change
shape.

### Tests added

In `aae::noxu_fold::tests` (in-tree, gated by `--features
noxu`):

* `empty_datastore_yields_empty_tree` -- fold over an empty
  datastore leaves every root at zero.
* `fold_inserts_every_primary_record` -- 1000 puts; fold
  yields 1000 leaves total across the tree.
* `fold_skips_2i_records` -- a single put with a 2i index
  yields exactly one leaf (not three: primary + forward +
  reverse).
* `fold_matches_explicit_inserts_bit_for_bit` -- 256 puts
  driven through both paths (one tree built from explicit
  `Tree::insert(bucket, key, value, 0)`, one from
  `build_from_noxu_fold`) produce identical roots and
  segments.

### Scheduler integration heuristic

The brief asks us to wire the fold path into the scheduler so
that "when the configured datastore is `NoxuDatastore` AND
the snapshot is missing/corrupt, prefer `build_from_noxu_fold`
over `rebuild_from_api`." The current AAE module is
self-contained and does not own a `Datastore` reference; the
caller (the dynomited integration in a follow-up slice) is
expected to:

1. `Tree::load_snapshot(cfg.snapshot_path())` first.
2. On `Err`:
   * If the configured datastore is `NoxuDatastore` AND the
     `noxu` feature is enabled, call
     `tree.build_from_noxu_fold(&ds)`.
   * Otherwise, fall back to a public-`Datastore`-API
     rebuild (out of scope for this slice; the API
     `Datastore::list_keys_stream` plus `riak_get` already
     exist for an embedder to walk).

This keeps the AAE module decoupled from the Datastore enum
the embedder picks, while preserving the heuristic the brief
asks for. The dynomited integration journal entry will
reference back to this design.
