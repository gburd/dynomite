## C-verification checks performed

The Stage 6 / Stage 7 reviews emphasised that the lead's briefs
contained factual errors about the C source. I re-verified each
claim about the C source against the actual C lines before
implementing. The independent checks I ran:

1. **`MAX_CONN_QUEUE_SIZE 20000`** -
   `_/dynomite/src/dyn_connection.h` line 47: `#define
   MAX_CONN_QUEUE_SIZE 20000`. Verified.
2. **Backoff doubling** -
   `_/dynomite/src/dyn_connection_pool.c::conn_pool_notify_conn_errored`
   lines 184-200: `cp->current_timeout_sec = 2 *
   cp->current_timeout_sec`, capped by `cp->max_timeout_sec`,
   initial value 1s via `MIN_WAIT_BEFORE_RECONNECT_IN_SECS`.
   Verified.
3. **`conn_pool_get` round-robin via `tag % count`** -
   `_/dynomite/src/dyn_connection_pool.c::conn_pool_get` line
   124: `array_get(&cp->active_connections, (uint32_t)tag %
   array_n(&cp->active_connections))`. Returns NULL when the
   slot's conn is not `connected`. Verified.
4. **`conn_pool_connected` resets failure_count and timeout** -
   `_/dynomite/src/dyn_connection_pool.c` line 207: `cp->failure_count
   = 0; cp->current_timeout_sec = 0;`. Verified.
5. **Auto-eject is per-server, not per-pool** -
   `_/dynomite/src/dyn_server.c::datastore_check_autoeject` lines
   317-333: per-`datastore` `next_retry_ms` field; the C engine
   evaluates it before every `get_datastore_conn`. Verified;
   the Rust port lifts the policy into a standalone
   `AutoEject` so the Stage-10 cluster code can reuse the same
   policy for peers.
6. **`conn_listen` does not call `setsockopt(IPV6_V6ONLY)`** -
   `_/dynomite/src/dyn_connection.c::conn_listen` lines 167-220:
   only `dn_set_reuseaddr`, `dn_set_nonblocking`,
   `dn_set_keepalive`. The Rust port adds explicit
   `Socket::set_only_v6(false)` to make the dual-stack bind
   deterministic across platforms; documented as a deviation
   in `docs/parity.md`.
7. **`conn_ops` vtable shape** - `_/dynomite/src/dyn_connection.h`
   lines 70-89: the 15-entry vtable with the slots I quoted.
   Verified that PROXY / DNODE_PEER_PROXY use a vtable with
   only `recv`, `close`, `ref`, `unref`, and
   `conn_cant_handle_response` populated; the others are
   `NULL`.
8. **`req_filter` shape** - `_/dynomite/src/dyn_client.c::req_filter`
   lines 296-330: empty / quit / dyno-config branches. The
   Stage 9 port consolidates into the parser-result `Noop`
   arm; the synthesized OK response on quit and the
   dyno-config side effect are deferred to Stage 10 because
   they require dispatcher hooks.
9. **`_conn_get` always allocates a fresh AES key** -
   `_/dynomite/src/dyn_connection_internal.c::_conn_get` lines
   78-87: `generate_aes_key()` is called unconditionally, even
   for non-secured peer conns, and the key is copied into
   `conn->aes_key`. The Rust port follows this shape: every
   `Conn` has an optional `aes_key` slot, and the Stage 6
   `Crypto::generate_aes_key` is invoked by the Stage-10
   handshake driver before installing the key. The Stage 9
   driver does not pre-generate keys because Stage 9 does not
   yet wire the handshake message.
10. **`dnode_rsp_send_next` prepend logic** -
    `_/dynomite/src/dyn_dnode_client.c::dnode_rsp_send_next` lines
    272-340: the C code prepends a header mbuf via
    `mbuf_insert_head` after calling `dmsg_write` with
    `msg_length(rsp)`. Verified the Rust port emits the
    header and payload as two separate writes through the
    transport, which is observably equivalent on the wire.
11. **The C engine uses `dictAdd`, not `dictReplace`** -
    `_/dynomite/src/dyn_client.c::req_forward` line 525:
    `dictAdd(c_conn->outstanding_msgs_dict, &req->id, req)`.
    The Rust port uses `HashMap::insert` (overwrite), which
    is observably equivalent because the C engine never adds
    a duplicate key (msg ids are monotonic per-connection).

## Review response

Addresses the four blockers and three warnings from the Stage 9
review (`docs/journal/review-stage-9-7cef8df1.md`).

### Blockers

* **Blocker 1 - decrypt_dnode_payload was a no-op stub.** Now
  calls `Crypto::aes_decrypt(payload, conn.aes_key())` when the
  dnode header marks the payload as encrypted. Failures collapse
  to one opaque `NetError::Dnode` so peers cannot distinguish
  bad-padding from decryption-failed (the Stage 6 review's
  padding-oracle follow-up). Plaintext peers (header without the
  encrypted flag) continue to pass through unchanged. Regression
  test `dnode_client_decrypt_round_trip` in
  `tests/stage_09_net.rs` round-trips a real AES-encrypted Redis
  payload through the inbound peer-client driver. Commit
  `e42b41b`.

* **Blocker 2 - drive_dnode_parser dropped parsed peer
  requests.** Now invokes
  `handler.dispatcher().dispatch(msg, responder)` and routes the
  outcome the same way `client_loop` does: Pending and Drop are
  no-ops, Inline and Error are forwarded immediately through the
  per-connection responder channel. The peer-client loop also
  writes any envelope received on its responder channel back
  over the inbound transport so dispatcher-produced responses
  reach the originating peer. `ClientHandler` gained
  `dispatcher()` and `response_tx()` accessors so the dnode
  driver can share the dispatch contract without breaking
  encapsulation. `client_loop` now carries the consumed request
  bytes into `msg.mbufs()` before dispatch so a forwarding
  dispatcher can replay them to a backend without re-encoding.
  Regression test `dnode_peer_round_trip` in
  `tests/stage_09_net.rs` asserts the dispatcher actually sees
  the parsed peer request. Commit `e42b41b`.

* **Blocker 3 - QUIC end-to-end test claim was unbacked.**
  Picked option (a) from the brief. Added
  `crates/dynomite/tests/stage_09_quic.rs` with a single
  `quic_loopback_round_trip` test that generates a fresh
  self-signed cert at startup via the pure-Rust `rcgen` crate
  (no checked-in fixture, no openssl CLI), spawns a QuicListener
  on `127.0.0.1:0`, dials it, exchanges a 32-byte payload across
  the QUIC stream in both directions, and asserts equality. The
  test uncovered two real bugs in the driver pump:
  - `stream_send` before `is_established()` returned
    `StreamLimit`, which the driver treated as fatal. Fixed by
    buffering app bytes locally until the handshake completes.
  - The default `conn.timeout()` cadence (multiple seconds) made
    the loop too sluggish for an interactive proxy. Capped the
    sleep arm at 10 ms.
  Both fixes have rationale in the commit message and a
  parity.md entry under Deviations. The QUIC + crypto
  coexistence link clash (originally blocking this combination)
  is a non-issue now that crypto migrated to RustCrypto;
  documented as resolved in `blocked.md`. Workspace dev-deps
  gain `rcgen 0.13`; `time` was pinned to `0.3.41` /
  `time-core 0.1.4` to keep MSRV at 1.85.
  Stage 9 also fixed `DnodeProxy::run`'s factory contract so
  the per-peer `(tx, rx)` is actually wired into the handler;
  previously the `tx` was orphaned, leaving the inbound driver
  unable to surface a dispatcher response. Commit `cf44720`.

* **Blocker 4 - no CLIENT-SERVER or DNODE round-trip tests.**
  Added two integration tests that drive real bytes across
  tokio TCP loopback connections (not just FSM unit tests):
  - `client_server_redis_round_trip` connects a real TCP client
    to a `Proxy` whose `ForwardingDispatcher` reconstructs the
    parsed `Msg`'s mbuf chain into the original wire bytes and
    pushes them through a `ServerConn` outbound driver to a mock
    Redis backend on a separate `127.0.0.1:0` port. The mock
    backend asserts the SET command bytes round-trip exactly;
    the proxy then returns the backend's `+OK\r\n` to the
    client, which is also asserted.
  - `dnode_peer_round_trip` opens a TCP loopback pair, drives
    one end with `DnodeServerConn::run` (outbound) and the other
    with `dnode_client_loop` (inbound) plus a
    `RecordingDispatcher`. Asserts the parsed request reaches
    the dispatcher and its bytes match the originally-sent
    payload.
  Commit `dae91e0`.

### Warnings

* **Warning 1 - Proxy::run missing set_nodelay.** The
  accept loop now calls `sock.set_nodelay(true)` (errors are
  non-fatal). Regression test `proxy_run_sets_tcp_nodelay`
  pins the contract: after `set_nodelay(true)` on a fresh
  accepted `TcpStream`, `socket2::SockRef::nodelay()` reports
  `true`. Commits `e59113f` (code) and `dae91e0` (test).

* **Warning 2 - QuicTransport::poll_write hot-spinned on
  Full.** Replaced `try_send` with `tokio_util::sync::PollSender`.
  `poll_write` now calls `tx.poll_reserve(cx)`: on `Ready(Ok)`
  the bytes are sent via `send_item`, on `Pending` the caller's
  waker is registered and we return `Poll::Pending`. No more
  short writes. Commit `e59113f` (code; same commit folded the
  rustdoc cleanup).

* **Warning 3 - port-acknowledgement comments throughout net/.**
  Rewrote rustdoc / comments in net/, proto/dnode.rs, msg/,
  crypto/, examples/, and stage_03 / stage_07 integration tests
  to describe behaviour semantically without naming C symbols.
  Cross-references to the C reference engine remain in
  `docs/parity.md` only. Tightened
  `scripts/check_no_port_comments.sh` to flag
  `Mirrors \`dyn_`, `Mirrors the \`dyn_`, `Mirror[s]? .* dyn_`,
  `C reference engine`, and `dyn_[a-z_]+\.c`. The script now
  produces no output across the workspace. Commit `e59113f`.
