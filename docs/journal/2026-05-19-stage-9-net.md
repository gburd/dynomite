## C-verification checks performed

The Stage 6 / Stage 7 reviews emphasised that the lead's briefs
contained factual errors about the C source. I re-verified each
claim about the C source against the actual C lines before
implementing. The independent checks I ran:

1. **`MAX_CONN_QUEUE_SIZE 20000`** -
   `_/dynomite/src/dyn_connection.h` line 47: `#define
   MAX_CONN_QUEUE_SIZE 20000`. Verified.
2. **Backoff doubling** -
   `_/dynomite/src/dyn_connection_pool.c::conn_pool_notify_conn_errored`
   lines 184-200: `cp->current_timeout_sec = 2 *
   cp->current_timeout_sec`, capped by `cp->max_timeout_sec`,
   initial value 1s via `MIN_WAIT_BEFORE_RECONNECT_IN_SECS`.
   Verified.
3. **`conn_pool_get` round-robin via `tag % count`** -
   `_/dynomite/src/dyn_connection_pool.c::conn_pool_get` line
   124: `array_get(&cp->active_connections, (uint32_t)tag %
   array_n(&cp->active_connections))`. Returns NULL when the
   slot's conn is not `connected`. Verified.
4. **`conn_pool_connected` resets failure_count and timeout** -
   `_/dynomite/src/dyn_connection_pool.c` line 207: `cp->failure_count
   = 0; cp->current_timeout_sec = 0;`. Verified.
5. **Auto-eject is per-server, not per-pool** -
   `_/dynomite/src/dyn_server.c::datastore_check_autoeject` lines
   317-333: per-`datastore` `next_retry_ms` field; the C engine
   evaluates it before every `get_datastore_conn`. Verified;
   the Rust port lifts the policy into a standalone
   `AutoEject` so the Stage-10 cluster code can reuse the same
   policy for peers.
6. **`conn_listen` does not call `setsockopt(IPV6_V6ONLY)`** -
   `_/dynomite/src/dyn_connection.c::conn_listen` lines 167-220:
   only `dn_set_reuseaddr`, `dn_set_nonblocking`,
   `dn_set_keepalive`. The Rust port adds explicit
   `Socket::set_only_v6(false)` to make the dual-stack bind
   deterministic across platforms; documented as a deviation
   in `docs/parity.md`.
7. **`conn_ops` vtable shape** - `_/dynomite/src/dyn_connection.h`
   lines 70-89: the 15-entry vtable with the slots I quoted.
   Verified that PROXY / DNODE_PEER_PROXY use a vtable with
   only `recv`, `close`, `ref`, `unref`, and
   `conn_cant_handle_response` populated; the others are
   `NULL`.
8. **`req_filter` shape** - `_/dynomite/src/dyn_client.c::req_filter`
   lines 296-330: empty / quit / dyno-config branches. The
   Stage 9 port consolidates into the parser-result `Noop`
   arm; the synthesized OK response on quit and the
   dyno-config side effect are deferred to Stage 10 because
   they require dispatcher hooks.
9. **`_conn_get` always allocates a fresh AES key** -
   `_/dynomite/src/dyn_connection_internal.c::_conn_get` lines
   78-87: `generate_aes_key()` is called unconditionally, even
   for non-secured peer conns, and the key is copied into
   `conn->aes_key`. The Rust port follows this shape: every
   `Conn` has an optional `aes_key` slot, and the Stage 6
   `Crypto::generate_aes_key` is invoked by the Stage-10
   handshake driver before installing the key. The Stage 9
   driver does not pre-generate keys because Stage 9 does not
   yet wire the handshake message.
10. **`dnode_rsp_send_next` prepend logic** -
    `_/dynomite/src/dyn_dnode_client.c::dnode_rsp_send_next` lines
    272-340: the C code prepends a header mbuf via
    `mbuf_insert_head` after calling `dmsg_write` with
    `msg_length(rsp)`. Verified the Rust port emits the
    header and payload as two separate writes through the
    transport, which is observably equivalent on the wire.
11. **The C engine uses `dictAdd`, not `dictReplace`** -
    `_/dynomite/src/dyn_client.c::req_forward` line 525:
    `dictAdd(c_conn->outstanding_msgs_dict, &req->id, req)`.
    The Rust port uses `HashMap::insert` (overwrite), which
    is observably equivalent because the C engine never adds
    a duplicate key (msg ids are monotonic per-connection).

## Ambiguities

None new. The Stage 7 ambiguity about parser chain spanning
remains valid: the Stage 9 driver feeds the parser
contiguous bytes assembled in a `Vec<u8>` rather than walking
the mbuf chain directly; the underlying parser is the same.

## Deviations recorded

1. `Stage 9: ConnPool ownership replaces the C ref/unref dance`
   - Rust ownership replaces the manual ref/unref calls; the
     observable behaviour is identical.
2. `Stage 9: auto_eject lifted out of dyn_server.c`
   - The shared policy is factored into `net::auto_eject` so
     the pool and the Stage 10 cluster code reuse one impl.
3. `Stage 9: QUIC + openssl-vendored static-link conflict`
   - The QUIC transport ships behind a feature flag; the
     end-to-end QUIC test is deferred until the workspace
     resolves the openssl-vendored vs quiche-boringssl link
     conflict.

  `decrypt_to_vec` errors to a single externally-visible
  variant in the DNODE handshake context) is **partially**
  addressed: the Stage 9 dnode_client driver wraps the
  decrypt result through `decrypt_dnode_payload`, which
  currently passes the bytes through unchanged. The full
  decrypt + error-collapse wiring lands when Stage 10 adds
  the handshake message handler.

  open (deferred to the cargo-fuzz crate creation pass).
