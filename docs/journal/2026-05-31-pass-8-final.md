## What's notable

**Memcache went from 67.28% (pass-7) to 99.68% (pass-8)** --
a +32 point improvement on the aggregate. Same hosts. The
delta is entirely the Bug C fix (start-host now kills a stale
backend container from a prior mode before bringing up the new
mode). All three captured hosts exceed 99.6% individually.

**Redis went from 95.17% (pass-7) to 98.55% (pass-8)**, +3.4
points. The retry-backoff + heredoc fix + per-mode rsync +
chaos-ops bug fixes all stacking up.

**Riak**: the per-mode rsync didn't fire in riak mode for
arnold/meh, so we only have floki data (84.99%). The data
plane itself was healthy on floki; the chaos coordinator's
post-mode rsync for the LAST mode (which is riak) is a
separate code path that bypasses the per-mode hook.

## Open follow-ups

1. **Riak post-mode rsync**: the riak-mode capture for
   arnold/meh failed silently. The post-mode rsync runs at
   the END of each mode, before next-mode startup wipes the
   remote. For the LAST mode there's no next-mode startup,
   so the post-mode rsync is implicitly the final teardown
   rsync; that one apparently timed out or didn't run. Fix:
   make the wrapper run an explicit post-mode-rsync after
   the LAST mode too.

2. **nuc operational state**: rsync still fails to nuc.
   Direct LAN rsync to 192.168.x is broken (Tailscale/LAN
   bridging issue); ProxyJump-via-arnold rsync works for
   small ssh commands but not the bandwidth-heavy bootstrap
   transfer. Workaround: pre-stage the noxu + dynomite
   trees on nuc by hand. Long-term: fix Tailscale-direct
   on nuc.

3. **arnold riak-mode start failure** (intermittent): when
   it does happen, the chaos-ops-fixes worker's
   START_HOST_STATS_TIMEOUT_SECS knob (default 30s) gives
   us 30 attempts at 1s instead of 30 at 0.2s, but the
   underlying race is dynomited's stats listener
   sometimes binding slowly. Investigate when it recurs.
