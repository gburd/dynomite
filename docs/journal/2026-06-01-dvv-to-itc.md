# 2026-06-01 -- DvvSet/Vclock -> Interval Tree Clocks

Causality clock migration. The dyniak per-key causality tracker
moves from a Dotted Version Vector Set (and the legacy `Vclock`
shadow type) to an Interval Tree Clock
([`dyniak::datatypes::Itc`]). The eight ITC operations from
Almeida, Baquero, Fonte (2008) -- seed, fork, join, peek, event,
leq, send, receive -- are exposed as inherent methods on `Itc`.

## Rationale

ITC scales with the population of currently-live actors, not
with the population of every actor that has ever participated.
DVVSet (and classic VVs) carry a per-actor entry forever; under
dyniak's dynamic-membership cluster model that grows
unboundedly. ITC's `fork` / `join` keeps the stamp size
proportional to the live partition of the ownership tree, and
retired actors leave no residual cost.

## User-visible break

The Riak protobuf `DtFetchResp.context` and `DtUpdateReq.context`
opaque blob is now `Itc::encode()`-shaped instead of
DvvSet-shaped. Clients that round-trip the bytes opaquely (the
documented Riak contract) are unaffected. Clients that crack
the blob and decode it as a Riak DVV will fail to parse and need
to switch decoders. This is a deliberate divergence from
upstream Riak; it is recorded as the "Causality clock
divergence" entry under `docs/parity.md` Ambiguities and pinned
as the project's chosen interpretation.
