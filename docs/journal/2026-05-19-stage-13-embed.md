## Design choices

### Object-safe async via `BoxFuture`

`Datastore::dispatch`, `MetricsSink::emit`, and friends use
`BoxFuture<'a, Result<T, E>>` rather than the `async fn`
return-position trait shape Rust 1.75+ allows. Reason: object
safety is a hard requirement (the builder takes `Box<dyn
Datastore>` etc.); without `async_trait` on the workspace
dependency list the boxed-future pattern is the cleanest path.
Type alias `BoxFuture<'a, T>` is exposed publicly so embedders
write the same shape as the in-crate impls.

### In-process peer registry

`inject_request` needs to honour `DispatchPlan::Replicas`
without wiring a real dnode forward path through the
connection-pool / mbuf chain (that work belongs to the live
runtime stages, not the embed facade). A process-wide static
registry keyed on `dyn_listen` `SocketAddr` lets the embed
surface forward to a co-located peer's `Datastore` directly.
Production cross-process traffic is unchanged; the registry is
never consulted unless the target peer's address resolves to a
`SocketAddr` that another in-process server registered.

Recorded as a Deviation in `docs/parity.md`.

### `RustCryptoProvider` over `OpensslCryptoProvider`

The design doc names the default crypto provider
`OpensslCryptoProvider`. The workspace migrated to RustCrypto in
Stage 6 (`docs/journal/2026-...-stage-06-crypto.md`); the embed
default keeps the type name aligned with the actual backend. The
trait surface is identical.

### `LoggingMetricsSink` over `PrometheusMetricsSink`

Same reasoning as Stage 13's brief: shipping a Prometheus
exporter would add a workspace dependency, which is governed by
PLAN.md Section 2 plus a journal entry. `LoggingMetricsSink` is
dependency-free, exercises the trait surface, and documents the
Prometheus shape in the rustdoc + design doc for the embedder
who needs it. Recorded as a Deviation.

### Custom-transport example uses `tokio::io::duplex`

`embedded_with_custom_transport.rs` shows the `Transport` trait
shape with a dependency-free in-memory `DuplexStream`. The
inline doc note points at `tokio_rustls::TlsStream` for
production wiring; the trait surface does not change.

### Builder validates `ConfPool` directly

`ServerBuilder::build` calls `ConfPool::apply_defaults` +
`ConfPool::validate(pool_name)` rather than serialising the pool
to YAML and re-parsing through `Config`. The validation surface
is identical (the same helper backs `Config::validate`); the
direct path avoids an asymmetric serde shape on `ConfDynSeed`
(serialises as `pname` only, deserialises full form).

## SIGHUP / `dynomite_reload_conf` parity

The C reference's `dynomite_reload_conf` (in
`_/dynomite/src/dynomite.c`) re-reads the YAML from disk and
swaps the global pool reference under a lock. Stage 13 mirrors
this with `ServerHandle::reload(Config)`:

1. Run `Config::validate` first; on failure, leave the running
   pool untouched.
2. Bump a generation counter and emit
   `ServerEvent::ConfigReloaded { generation }`.
3. Yield once so the broadcast send is observable on the calling
   task's scheduler before the function returns.

The full pool-rebuild path (peer-table swap, ring rebuild) is
deferred to the runtime-loop stage; the validation + event
contract that the embed surface promises today is the long-lived
contract under SemVer.

## Review response

The `review/stage-13` agent (87ace611) returned
REQUEST_CHANGES against the SemVer-fragile public surface and
the accept-loop / custom-transport-example contracts. The
follow-up commits on this branch address the required-change
list verbatim:

* **Change 1 - non_exhaustive on public enums.** Added
  `#[non_exhaustive]` to `EmbedError`, `DatastoreError`,
  `MetricsError`, `CryptoProviderError`, `Protocol`,
  `ConnRoleTag`, `CloseReason`, `PeerDownReason`. (`ServerEvent`
  was already non-exhaustive.) Adding a variant in a future
  stage is now a SemVer-minor change.

* **Change 2 - sealed `ServerHooks`.** Fields are now
  `pub(crate)`; the struct carries `#[non_exhaustive]`. Read-back
  is through new `datastore()`/`seeds()`/`crypto()`/`metrics()`
  accessor methods. Construction is routed through
  `ServerBuilder` only (no in-tree caller built `ServerHooks`
  directly).

* **Change 3 - hidden internal-type leaks.** `Server::cluster_pool()`,
  `ServerBuilder::conf_pool_mut()`, the `pub fn shutdown_signal()`
  and `pub fn _instant_now()` escape hatches, the `pub type
  SharedDatastore` alias, and the `LegacySeedsAdapter<T: RawSeeds>`
  generic wrapper have all been removed or downgraded to
  `pub(crate)`. The `RawSeeds` trait alias remains in scope
  inside `hooks.rs` because the `DnsSeedsProvider` /
  `FloridaSeedsProvider` blanket impls call its `get_seeds`
  method.

* **Change 4 - accept_loop documented + warn-logged.** The
  embedded accept loop now emits a `tracing::warn!` event on
  every accept stating that the embedded server does not yet
  forward to the dispatcher and pointing the caller at
  `ServerHandle::inject_request` (in-process) or the
  `dynomited` binary (cross-process). The contract is also
  spelled out in the rustdoc on `Server::start`. Wiring the
  accept loop to `ClusterDispatcher` is deliberately deferred:
  the proxy module landed in Stage 9 and the embedded
  surface should not relitigate that wiring.

* **Change 5 - custom-transport example rewritten as a
  sketch.** Renamed
  `embedded_with_custom_transport.rs` ->
  `embedded_custom_transport_sketch.rs`. The example now
  implements the public `dynomite::embed::Transport` trait
  (it previously only satisfied `AsyncRead + AsyncWrite + Send +
  Unpin` plus an inherent `role()` method) and the doc header
  states that the sketch demonstrates the trait shape only:
  wiring a `Box<dyn TransportListener>` into the builder is
  tracked as Stage 14b.

* **Change 6 - three contract divergences recorded.** New
  entries in `docs/parity.md` under "Stage 13: deviations and
  follow-ups":
  1. `Server::builder()` takes a `pool_name` argument; ergonomic
     reasons + a new `ServerBuilder::with_pool_name(name)`
     setter and a `ServerBuilder::default()` impl that pre-fills
     `"dyn_o_mite"`.
  2. `Server::start()` is `async fn -> Result<ServerHandle,
     EmbedError>` because bind failures must be surfaced.
  3. `Datastore` is single-method (`dispatch`) in v0.1; the
     two-tier `connect()` + `DatastoreConn` shape from the
     design page is collapsed. The design page should be
     updated to reflect the v0.1 contract.

* **Change 7 - `tokens_str` swallow-with-log.** The setter now
  emits a `tracing::warn!` event on parse failure (so a typo is
  observable in logs) and returns the builder unchanged so the
  chain stays ergonomic. A future SemVer-major release may
  switch to a `Result`-returning shape; the swallow-with-log
  behaviour is recorded as the v0.1 contract.

* **Change 8 - `EventBus::send` is `pub(crate)`.** Only the
  engine publishes events; embedding programs subscribe via
  `EventBus::subscribe` and consume via `EventStream::recv` /
  `EventStream::try_recv`. The doctest on `EventStream::recv`
  was rewritten to demonstrate the closed-stream shape
  (publishing requires the engine, which is not in scope for a
  unit doctest).

The doctest gaps the reviewer flagged on `ServerHandle::shutdown`,
`reload`, `inject_request`, `stats`, `describe_stats`, `peers`,
`datacenters`, `ring`, `crypto_provider`, `listen_addr`,
`dyn_listen_addr` are deferred to the project-wide doctest
cleanup pass tracked in `docs/journal/blocked.md`; the review
response explicitly scoped them out.
